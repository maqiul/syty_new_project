## 一、背景与现状

### 1.1 系统现状

当前系统（V1.0）已实现基础的穿线订单管理，但存在**四大痛点**：

| 痛点 | 具体表现 | 影响 |
|---|---|---|
| **数据割裂** | `player`（羽毛球）和 `tennis_player`（网球）两套表完全独立，球拍/线表同理 | 同一客户打两种球要建两个档案；查询/统计需 UNION 两套逻辑 |
| **库存黑洞** | `shop_string` 表有 `stock` 字段但无流水记录，扣库存无迹可查 | 库存对不上不知道谁干的；无入库/出库记录 |
| **财务盲区** | `stringing_order` 无支付状态、无支付流水，收了多少钱、欠了多少钱全靠人脑记 | 老板不知道应收多少；熟客挂账无法追踪 |
| **绩效空白** | 订单未关联穿线师，无法统计谁穿了多少钱、提成怎么算 | 穿线师工资算不清；C 端打印小票无法体现穿线师 |

### 1.2 V1.1 核心目标

> **一句话：羽网合并、进销存闭环、算清账、算清提成。**

## 二、模块一：数据模型重构（羽网合并）

### 2.1 变更概述

| 旧表 | 新表 | 动作 |
|---|---|---|
| `player` + `tennis_player` | `player`（统一） | 合并 + 新增 `sport_type` |
| `racket` + `tennis_racket` | `racket`（统一） | 合并 + 新增 `sport_type` |
| `string_info` + `tennis_string_info` | `string_info`（统一） | 合并 + 新增 `sport_type` |
| `tennis_order` | ~~废弃~~ | 合并到 `stringing_order`，已有 `sport_type` 无需新建表 |

**核心原则：** 保留羽毛球表为主体，网球字段通过 `sport_type` + 扩展字段兼容，不做表分拆。

### 2.2 统一 `player` 表结构

```sql
-- ============================================
-- 统一球员表（原 player + tennis_player 合并）
-- ============================================
CREATE TABLE player_new (
    id              BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    name            VARCHAR(50)  NOT NULL,
    phone           VARCHAR(20)  NOT NULL DEFAULT '',
    gender          SMALLINT     DEFAULT 0 COMMENT '0-未知 1-男 2-女',
    sport_type      VARCHAR(10)  NOT NULL DEFAULT 'BADMINTON' COMMENT '运动类型: BADMINTON-羽毛球 TENNIS-网球 DUAL-双修',
    -- 网球专属字段（羽毛球客户可为空）
    ranking         VARCHAR(50)  DEFAULT '' COMMENT '网球排名（仅网球有效）',
    -- 通用字段
    photo_url       VARCHAR(500) DEFAULT '',
    skill_level     VARCHAR(20)  DEFAULT '' COMMENT '水平等级',
    progress_notes  TEXT         DEFAULT '' COMMENT '近期进步/备注',
    remark          VARCHAR(500) DEFAULT '',
    tenant_id       BIGINT       DEFAULT NULL COMMENT '所属租户',
    -- 审计字段
    deleted         SMALLINT     NOT NULL DEFAULT 0,
    created_at      TIMESTAMP    NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at      TIMESTAMP    NOT NULL DEFAULT CURRENT_TIMESTAMP
);
COMMENT ON TABLE player_new IS '统一球员表（羽毛球+网球）';
CREATE INDEX idx_player_phone ON player_new(phone);
CREATE INDEX idx_player_tenant ON player_new(tenant_id);
CREATE INDEX idx_player_sport ON player_new(sport_type);
```

**变更点说明：**
- `sport_type` 区分客户类型：`BADMINTON`、`TENNIS`、`DUAL`（两种都打）
- `ranking` 从网球表迁移，羽毛球客户此字段为空即可
- 新增 `tenant_id`（原表没有），支持多租户隔离
- `phone` 设为必填（原 `player` 没有 NOT NULL 约束），同一租户内手机号唯一索引

## 三、模块二：库存管理闭环（进销存）

### 3.1 现状分析

当前 `shop_string` 表：
```
shop_string: id, tenant_id, shop_id, string_id, price, stock, min_stock_alert, ...
```

**问题：**
- `stock` 字段直接改，没有任何操作记录
- 无法追溯"谁在什么时候扣了多少库存"
- 没有入库记录，采购了一批线无法录入系统
- 库存为负数时没有拦截

### 3.2 `stock_log` 流水表设计

```sql
-- ============================================
-- 库存流水表（核心！所有库存变动必须经过此表）
-- ============================================
CREATE TABLE stock_log (
    id              BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    tenant_id       BIGINT       NOT NULL COMMENT '租户ID',
    shop_id         BIGINT       NOT NULL COMMENT '店铺ID',
    string_id       BIGINT       NOT NULL COMMENT '线材ID',
    
    -- 变动信息
    change_type     VARCHAR(20)  NOT NULL COMMENT '变动类型: PURCHASE_IN-采购入库 ORDER_OUT-订单扣减 MANUAL_ADJUST-手动调整 RETURN_IN-退货入库 TRANSFER-调拨',
    quantity        INTEGER      NOT NULL COMMENT '变动数量(正数为入库,负数为出库)',
    before_quantity INTEGER      NOT NULL COMMENT '变动前库存',
    after_quantity  INTEGER      NOT NULL COMMENT '变动后库存',
    
    -- 关联业务
    order_id        BIGINT       DEFAULT NULL COMMENT '关联订单ID(订单扣减时填写)',
    order_no        VARCHAR(50)  DEFAULT '' COMMENT '关联订单号',
    
    -- 操作信息
    operator_id     BIGINT       DEFAULT NULL COMMENT '操作人ID',
    operator_name   VARCHAR(50)  DEFAULT '' COMMENT '操作人姓名',
    remark          VARCHAR(200) DEFAULT '' COMMENT '备注(如采购单号、调整原因)',
    
    created_at      TIMESTAMP    NOT NULL DEFAULT CURRENT_TIMESTAMP
);
COMMENT ON TABLE stock_log IS '库存流水表';
CREATE INDEX idx_stock_log_shop ON stock_log(shop_id);
CREATE INDEX idx_stock_log_string ON stock_log(string_id);
CREATE INDEX idx_stock_log_order ON stock_log(order_id);
CREATE INDEX idx_stock_log_created ON stock_log(created_at);
```

### 3.3 业务流程

#### 3.3.1 订单完成自动扣减库存

```
┌─────────────────────────────────────────────────────────┐
│                    订单完成触发扣库存                       │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  1. 订单状态变更为 COMPLETED（已完成）                      │
│     ↓                                                   │
│  2. 检查订单是否有 main_string_id / cross_string_id       │
│     ↓                                                   │
│  3. 查询 shop_string 表获取当前库存                         │
│     ↓                                                   │
│  4. 库存不足？                                             │
│     ├── 是 → 记录 WARN 日志，继续扣减（允许负库存，但标记异常）  │
│     └── 否 → 继续                                        │
│     ↓                                                   │
│  5. 在 stock_log 插入两条流水（主线/横线各一条）              │
│     ├── change_type = 'ORDER_OUT'                        │
│     ├── quantity = -1                                    │
│     ├── before_quantity = 当前库存                         │
│     ├── after_quantity = 当前库存 - 1                     │
│     ├── order_id / order_no 关联订单                       │
│     └── operator_id = 订单创建人                          │
│     ↓                                                   │
│  6. UPDATE shop_string SET stock = stock - 1             │
│     ↓                                                   │
│  7. 检查 stock < min_stock_alert？                         │
│     └── 是 → 推送库存预警通知（前端标红）                     │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

**关键点：**
- 第5步和第6步**必须在同一事务中执行**，要么都成功，要么都失败
- 一条订单可能消耗主线和横线，所以最多产生两条 `stock_log`
- 如果订单没有指定具体线材（客户自带线），则不扣库存

#### 3.3.2 采购入库

```
┌─────────────────────────────────────────────────────────┐
│                       采购入库流程                         │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  1. 前端进入「库存管理 → 采购入库」页面                      │
│     ↓                                                   │
│  2. 选择店铺、线材、输入采购数量                             │
│     ↓                                                   │
│  3. 查询 shop_string 是否存在该店铺+线材记录                │
│     ├── 不存在 →  INSERT 一条新记录, stock = 采购数量       │
│     └── 存在 →  继续                                     │
│     ↓                                                   │
│  4. 在 stock_log 插入流水                                  │
│     ├── change_type = 'PURCHASE_IN'                      │
│     ├── quantity = 采购数量(正数)                          │
│     ├── before_quantity = 当前库存                         │
│     ├── after_quantity = 当前库存 + 采购数量               │
│     └── operator_id = 当前登录用户                         │
│     ↓                                                   │
│  5. UPDATE shop_string SET stock = stock + 采购数量         │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

#### 3.3.3 手动调整（盘盈/盘亏）

```
前端操作：库存管理 → 选择线材 → 点击「调整」
输入：调整数量（正/负）、调整原因
```

### 3.6 事务保障（给老刘的叮嘱）

```java
// 伪代码示例 - 订单完成时扣库存
@Transactional(rollbackFor = Exception.class)
public void completeOrder(Long orderId) {
    // 1. 更新订单状态
    order.setStatus(ORDER_STATUS_COMPLETED);
    orderMapper.updateById(order);
    
    // 2. 扣减主线库存
    if (order.getMainStringId() != null) {
        deductStock(order, order.getMainStringId());
    }
    
    // 3. 扣减横线库存
    if (order.getCrossStringId() != null && !order.getCrossStringId().equals(order.getMainStringId())) {
        deductStock(order, order.getCrossStringId());
    }
}

private void deductStock(StringingOrder order, Long stringId) {
    // 使用 SELECT FOR UPDATE 防止并发问题
    ShopString shopString = shopStringMapper.selectForUpdate(
        new QueryWrapper<ShopString>()
            .eq("shop_id", order.getShopId())
            .eq("string_id", stringId)
    );
    
    // 记录流水
    StockLog log = new StockLog();
    log.setChangeType("ORDER_OUT");
    log.setQuantity(-1);
    log.setBeforeQuantity(shopString.getStock());
    log.setAfterQuantity(shopString.getStock() - 1);
    log.setOrderId(order.getId());
    log.setOrderNo(order.getOrderNo());
    stockLogMapper.insert(log);
    
    // 更新库存
    shopString.setStock(shopString.getStock() - 1);
    shopStringMapper.updateById(shopString);
}
```

## 四、模块三：财务体系搭建（算清账）

### 4.1 现状分析

当前 `stringing_order` 有金额字段但无支付概念：
```
main_price, cross_price, labor_price, totalPrice -- 只有"应该收多少"
-- 没有：实际收了多少、怎么收的、是否已结清
```

### 4.2 `stringing_order` 表改造

```sql
-- 新增支付相关字段到 stringing_order
ALTER TABLE stringing_order ADD COLUMN pay_status     VARCHAR(20) NOT NULL DEFAULT 'UNPAID' COMMENT '支付状态: UNPAID-未付 PARTIAL-部分支付 PAID-已结清';
ALTER TABLE stringing_order ADD COLUMN paid_amount    DECIMAL(10,2) DEFAULT 0 COMMENT '已支付金额';
ALTER TABLE stringing_order ADD COLUMN pay_method     VARCHAR(20) DEFAULT '' COMMENT '支付方式: CASH-现金 WECHAT-微信 ALIPAY-支付宝 BANK-银行卡 CREDIT-挂账/赊账';
ALTER TABLE stringing_order ADD COLUMN pay_time       TIMESTAMP DEFAULT NULL COMMENT '最后支付时间';
ALTER TABLE stringing_order ADD COLUMN discount_amount DECIMAL(10,2) DEFAULT 0 COMMENT '优惠金额';
ALTER TABLE stringing_order ADD COLUMN actual_amount  DECIMAL(10,2) DEFAULT NULL COMMENT '实际应收金额(total_price - discount_amount)';

COMMENT ON COLUMN stringing_order.pay_status IS '支付状态: UNPAID-未付 PARTIAL-部分支付 PAID-已结清 CREDIT-挂账';
COMMENT ON COLUMN stringing_order.pay_method IS '支付方式: CASH-现金 WECHAT-微信 ALIPAY-支付宝 BANK-银行卡 CREDIT-挂账/月结';
```

**支付状态流转规则：**
```
UNPAID（未付） ──全额支付──> PAID（已结清）
UNPAID（未付） ──部分支付──> PARTIAL（部分支付）
PARTIAL（部分）──补足剩余──> PAID（已结清）
任何状态 ──标记挂账──> CREDIT（挂账中）
CREDIT（挂账） ──客户还款──> PAID（已结清）
```

### 4.3 `payment_record` 支付流水表

```sql

-- ============================================
CREATE TABLE payment_record (

    -- 支付后状态
    paid_amount_after DECIMAL(10,2) NOT NULL COMMENT '支付后该订单累计已付金额',
    pay_status_after  VARCHAR(20) NOT NULL COMMENT '支付后状态',
    
    -- 操作信息
    operator_id     BIGINT       DEFAULT NULL COMMENT '收款人ID',
    operator_name   VARCHAR(50)  DEFAULT '' COMMENT '收款人姓名',
    remark          VARCHAR(200) DEFAULT '' COMMENT '备注',
    
    created_at      TIMESTAMP    NOT NULL DEFAULT CURRENT_TIMESTAMP
);
COMMENT ON TABLE payment_record IS '支付流水表';
CREATE INDEX idx_payment_order ON payment_record(order_id);
```

#### 5.3.2 默认提成方案（推荐）

| 规则 | 计算方式 | 说明 |
|---|---|---|
| **方案A（固定）** | 每支 15 元 | 简单粗暴，适合新手穿线师 |
| **方案B（百分比）** | 手工费 × 30% | 穿线师和店铺利益绑定 |
| **方案C（阶梯）** | 月穿 ≤ 50支: 10元/支; > 50支: 15元/支 | 鼓励多劳多得 |

**提成计算时机：** 订单完成时（`status` 变更为 `COMPLETED`）

```java
// 伪代码
public BigDecimal calculateCommission(StringingOrder order, CommissionRule rule) {
    switch (rule.getRuleType()) {
        case "FIXED":
            return rule.getFixedAmount();
        case "PERCENT":
            BigDecimal base;
            switch (rule.getPercentBase()) {
                case "LABOR": base = order.getLaborPrice(); break;
                case "TOTAL": base = order.getTotalPrice(); break;
                case "STRING": base = order.getMainPrice().add(order.getCrossPrice()); break;
                default: base = order.getLaborPrice();
            }
            return base.multiply(rule.getPercentRate()).divide(new BigDecimal("100"), 2, RoundingMode.HALF_UP);
        case "TIERED":
            // 查询该穿线师当月已完成订单数
            int monthlyCount = orderMapper.countCompletedByStringerAndMonth(order.getStringerId(), currentMonth);
            // 匹配阶梯区间
            return matchTier(monthlyCount, rule.getTierConfig());
        default:
            return BigDecimal.ZERO;
    }
}
```

## 九、给老刘/大刘的特别说明

1. **`stringing_order` 这张表是本次改动最多的**，新增了 9 个字段。旧数据全部有默认值，不影响现有订单。
2. **库存扣减是异步还是同步？** 建议同步（在订单完成的事务内），因为穿线订单量不大，并发风险低。如果后期上小程序高并发再改异步。
3. **提成计算在订单完成时写入 `commission` 字段**，是冗余存储。如果提成规则改了，历史订单的提成不追溯。这是正确的设计——历史账单不能改。
4. **PostgreSQL 的 JSONB 字段用于阶梯提成配置**，查询时可以直接在 SQL 中解析，无需反序列化。
5. **所有 `ALTER TABLE` 操作在 PostgreSQL 中几乎瞬时完成**（只改元数据），不用担心锁表时间。
